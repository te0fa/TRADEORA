# Tradeora - المرحلة الأولى: جمع وتخزين بيانات سوق الأسهم المصري (EGX)

مشروع **Tradeora** هو نظام متكامل ومحترف لجمع بيانات البورصة المصرية وتخزينها في قاعدة بيانات Supabase. تم تصميم بنية المشروع بطريقة مرنة ومجزأة (Modular Architecture) تتيح التوسع وإضافة مصادر بيانات أخرى مستقبلاً.

---

## 📁 هيكل المشروع

```text
tradeora/
│
├── config/
│   └── settings.py          # إدارة الإعدادات وقراءة متغيرات البيئة بمرونة
│
├── database/
│   ├── db.py                # تهيئة اتصال Supabase وإدارة العمليات (CRUD / UPSERT)
│   └── schema.sql           # مخطط الجداول، الفهارس، القيود، وبذر البيانات المبدئية
│
├── scrapers/
│   └── egx_scraper.py       # سحب بيانات الأسعار من البورصة المصرية ومحاذاة الأعمدة
│
├── services/
│   └── importer.py          # معالجة جودة البيانات، إدارة الشركات، والـ UPSERT للأسعار
│
├── logs/
│   └── egx_scraper.log      # سجل تشغيل العمليات التفصيلي والأخطاء
│
├── tests/
│   └── test_pipeline.py     # اختبارات برمجية شاملة لوحدات السحب ومعالجة البيانات
│
├── data/
│   └── prices.html          # المجلد المحلي لحفظ الملفات المفرغة أو النشرات
│
├── main.py                  # ملف التشغيل الرئيسي للمشروع عبر سطر الأوامر
├── .env.example             # نموذج لملف الإعدادات البيئية
├── .gitignore               # استثناء الملفات الحساسة والسجلات من المستودع
├── requirements.txt         # المكتبات والمتطلبات البرمجية المطلوبة
└── README.md                # دليل الاستخدام والتوثيق (هذا الملف)
```

## ⚙️ التثبيت والإعداد

### 1. تثبيت المتطلبات البرمجية
تأكد من استخدام إصدار **Python 3.12+**. قم بتثبيت المكتبات المطلوبة بتشغيل الأمر التالي:

```bash
pip install -r requirements.txt
```

### 2. إعداد ملف البيئة `.env`
قم بإنشاء ملف `.env` في المجلد الرئيسي للمشروع، وانسخ فيه إعدادات `.env.example` ثم استبدل القيم ببيانات اتصال **Supabase** الخاصة بك:

```env
# Supabase Configuration
SUPABASE_URL=https://your-project-id.supabase.co
SUPABASE_KEY=your-supabase-anon-or-service-role-key

# Scraper & Importer Configuration
LOG_LEVEL=INFO
DATA_FOLDER=./data
DEFAULT_SOURCE=egx_bulletin
```

### 3. إعداد الجداول في Supabase
انسخ محتوى الملف [`database/schema.sql`](file:///e:/TRADEORA/database/schema.sql) والصقه في **SQL Editor** داخل لوحة تحكم Supabase الخاصة بك، ثم قم بالتشغيل (Run) لإنشاء الجداول والقيود والمؤشرات.

## 🚀 تشغيل المشروع

تتم معالجة سحب وتخزين البيانات بطريقتين:

### أ. التشغيل التجريبي (Dry Run)
لتجربة سحب ومعالجة البيانات بالكامل والتحقق من صحتها وجودتها دون الاتصال الفعلي بقاعدة البيانات أو إدخال قيم حقيقية:
```bash
python main.py --dry-run
```

### ب. التشغيل الفعلي (Live Mode)
للاتصال الفعلي بقاعدة بيانات Supabase واستيراد البيانات وحفظها:
```bash
python main.py
```

### ج. التشغيل باستخدام ملف محلي (Local Parser Mode)
لتخطي مشاكل جدران الحماية الخارجية للموقع الرسمي، يمكنك تمرير ملف HTML (تم حفظه يدوياً أو آلياً) أو ملف Excel للنشرة:
```bash
python main.py --file data/prices.html
```

## 🛠️ تفاصيل هندسة النظام

### 1. الـ Scraper والـ Fallback
*   يحاول الـ Scraper الوصول إلى صفحة الأسعار المباشرة `prices.aspx` باستخدام مكتبة `requests` وترويسات تصفح حقيقية.
*   في حال وجود تحدي جدار الحماية (F5)، يطبق تشغيل متصفح خفي (Headless Selenium) مع تفعيل خيارات التخفي.
*   في حال استمرار الحظر، ينسحب الـ Scraper بمرونة للبحث في المجلد `data/` عن أي ملف أسعار محلي (`prices.html` أو `.xlsx` أو `.csv`) لمعالجته، مما يمنع توقف الخدمة.

### 2. التحقق من جودة البيانات (Data Quality Check)
قبل إدخال أي سعر في قاعدة البيانات، يمرر عبر مصفوفة تحقق تضمن:
*   سعر الفتح والإغلاق أكبر من صفر.
*   أعلى سعر >= أقل سعر.
*   سعر الإغلاق محصور بين أعلى سعر وأقل سعر.
*   الكمية والقيمة المتداولة أرقام غير سالبة (>= 0).
في حال فشل أي صف، يُسجل كـ **Warning** في السجلات وفي جدول `import_jobs` ولا تتوقف العملية للأسهم الأخرى.

## 📈 إضافة مصدر بيانات جديد مستقبلاً (مثل Mubasher)

بنية المشروع مهيأة بالكامل لاستيعاب مصادر بيانات أخرى باتباع الخطوات البسيطة التالية:

1.  **إضافة المصدر لقاعدة البيانات:** أضف اسم المعرف الجديد للمصدر في جدول `market_sources` في قاعدة البيانات (مثال: `mubasher`).
2.  **كتابة Scraper جديد:** أنشئ ملف scraper خاص بالخدمة في مجلد `scrapers` (مثال: `scrapers/mubasher_scraper.py`) بحيث يُرجع مصفوفة قواميس موحدة للأسهم.
3.  **إعداد ملف التشغيل:** قم باستيراد الـ Scraper الجديد في `main.py` ومرر اسم المصدر عبر متغير `--source` ووجهه للـ Importer ليقوم بعملية المعالجة والإدخال بنفس الطريقة ودون المساس بالمنطق البرمجي لقاعدة البيانات.

## ⚡ إدارة مساحة قاعدة البيانات والبيانات التاريخية (Database Size & Historical Data Policy)

> [!IMPORTANT]
> للحفاظ على بقاء قاعدة بيانات Supabase الحية ضمن النطاق المجاني المسموح به (بحد أقصى **0.5 جيجابايت**):

1. **البيانات اللحظية الحية (`source = 'TradingView'`):**
   * يتم تخزينها مؤقتاً لأغراض العرض اللحظي والمتابعة الآنية.
   * يتم تنظيفها وتصفيتها تلقائياً كل **7 أيام** عبر الـ Workflow الخاص بالجدولة لمنع التضخم.
2. **البيانات التاريخية للتدريب والتحليل (`tradingview_1h`, `tradingview_4h`, `tradingview_30m`):**
   * **يمنع تخزينها أو إبقاؤها بشكل دائم في قاعدة البيانات الحية.**
   * عند تشغيل أي عملية Backfill تاريخي جديدة (مثل سكربت `tv_backfill.py`):
     1. يجب فوراً وبمجرد اكتمال الجلب تصدير البيانات محلياً بصيغة Parquet إلى المجلد: `data/historical_exports/`.
     2. يجب تفريغ هذه السجلات التاريخية من قاعدة البيانات الحية **خلال نفس الأسبوع** عبر سكربت الحذف أو الـ SQL Editor لضمان عدم تراكمها.
     3. يتم تدريب النماذج الفنية وتحديثها محلياً بالاعتماد على ملفات Parquet المصدرة مباشرة عبر `train_model.py` دون استهلاك موارد قاعدة البيانات السحابية.
